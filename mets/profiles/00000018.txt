	<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#description_rules">description_rules:</a><br>
		<p>This
profile describes the rules and requirements for using METS as an
exchange format to support the collection and preservation of and
access to content in Australian digital repositories. It is a generic
profile not specific to a particular system or implementation, and
therefore does not, for example, contain any detail in the Technical
Requirements section. It is intended that repositories will develop and
register sub-profiles that detail the implementation-specific
requirements.</p>
		<p>Saying that a particular element or attribute
in this profile is not supported does not imply that it is prohibited.
However, it does imply that it may be ignored. In general,
implementations processing METS documents that conform to this profile
should attempt to preserve any undefined elements or attributes. </p>
		<p>A
conforming METS document represents a discrete object of interest for
the purposes of collection, preservation or access. An object will be
completely described in a conforming METS document, therefore in the
case of a composite object, the components will be described within the
document as well. </p>
		<p>An object (whether it is a single or
composite object) may be a component of another object. A conforming
METS document may therefore represent a whole object or a component, as
long as the component is a discrete object of interest with its own
descriptive metadata. </p>
		<p>Where an object is a component of
another object, a conforming METS document should express that
relationship through appropriate elements in the descriptive metadata
extension schema, i.e. self-referencing, either fully described in the
primary MODS, or in separate &lt;dmdSec&gt; elements.</p>
		<p>An
object or component described in a conforming METS document may be a
physical object that has not yet have been digitised or it may be a
digital object stored on a physical carrier. </p>
		<p>The amount of
data included in a conforming METS document will depend on the purpose
of the document. If the purpose is to submit an object to a repository
as an OAIS Submission Information Package (SIP), then a conforming METS
document must contain the files comprising the archival copy of the
object and its components. If the purpose is to transfer custody of an
object from one repository to another as an OAIS Archival Information
Package (AIP), it must also contain the supporting files and metadata
necessary for the object's long term preservation and access. If the
purpose is to deliver an object as an OAIS Dissemination Information
Package (DIP), a conforming METS document may contain only derivative
files or pointers to derivative files, along with sufficient metadata
to render or execute these files properly. In the case of complex
hierarchical objects, components may be represented by another METS
document and just referenced in the parent METS document.</p>
<p>The first example appended to this profile shows a submission
information package for a digitised still image with two files
comprising the archival copy of the object (master and co-master). The
&lt;dmdSec&gt; contains only identifiers for the still image and its
host collection as these can be used by the repository to retrieve a
full cataloguing record on ingest. There is no technical metadata for
the images themselves as this can be generated on ingest. However,
PREMIS &lt;object&gt;, &lt;event&gt; and &lt;agent&gt; metadata
elements are included because of the need to record the derivation
relationship, the hardware and software used in the creation events and
the type of manipulation used to create the co-master.</p>
		<p>The
second example appended to this profile shows a dissemination
information package that could be used to transfer a digitised still
image from the same collection from one repository to another. The
package includes a derivative master and two derivative files created
on ingest as well as the master and co-master. Seven events are
recorded, the creation events for each file and the ingestion events
for the master and co-master.</p>
		<p>Information encoded using the
extension schema listed in this profile should follow the appropriate
guidelines recommended by the extension schema's maintenance agencies.
For example, MODS encoded descriptive metadata should follow the MODS
Users Guidelines published by the Library of Congress' Network
Development and MARC Standards Office. Users of additional schema
should list these as extension schema in a sub-profile and follow the
appropriate guidelines. </p>
		<p>All available metadata should be
included for the archival copy of an object when custody is being
transferred. If no suitable extension schema covers all the metadata,
the extra elements can be encoded using a local extension schema.</p>
		<p>Controlled
vocabularies for elements that are mandatory in this profile and that
are not already included in METS or the registered extension schema are
listed in the Controlled Vocabularies section. Users of additional
elements constrained by controlled vocabularies should list these in
the Controlled Vocabularies section of a sub-profile.</p>
		<p>Values of "not applicable" and "unknown" are permitted for mandatory elements and attributes where data cannot be supplied.</p>

these must be captured in separate &lt;dmdSec&gt; elements and linked
via the GROUPID attribute.

</p>

<br>
				<p>Use of optional PREMIS metadata elements is not supported
by this profile but may be included if data is available. SIZE,
CHECKSUM and CHECKSUMTYPE are mandatory attributes for the METS
&lt;file&gt; element and so there is no need to include PREMIS
&lt;messageDigestAlgorithm&gt;, &lt;messageDigest&gt; or &lt;size&gt;

included in the information package. In this case set

&lt;relationshipSubtype&gt; to "derived from". (Note that there is no
need in this profile to specify the relationship in both directions.) </p>

sufficient for this purpose.

				<p>&lt;rightsMD&gt; is optional. If rights metadata is
already stored in the &lt;dmdSec&gt;, for example, it may not be

the METS object e.g. if the METS object is a digital surrogate for a
physical item. &lt;sourceMD&gt; may be used to describe source
materials between the original and current object where the source
materials are not digital objects.
</p>

For example, if a PDF was created from a Word document and the PDF is

there must be one &lt;digiprovMD&gt; for each event or agent.

provided for 'master' or archival objects, describing events (in
separate &lt;digiprovMD&gt; elements) which led to the creation of the
current object and its ingest in the transferring repository. This
includes changes to the object originally deposited (note that in
PREMIS, an object cannot be modified: an event which modifies an object
creates a new object) and changes of custody.
</p>

&lt;linkingObjectIdentifier&gt; matching the &lt;objectIdentifier&gt;

@ID="fileSec1"
<br>
				<p>A &lt;fileSec&gt; element containing at least one
&lt;fileGrp&gt; must be used if there are data files associated with

			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="fileSec3"
<br>
				<p>Each &lt;fileGrp&gt; must have a USE attribute with a
value from the Australian METS Profile &lt;fileGrp&gt; USE attribute
controlled vocabulary, which is a list of values that may be used to
sort files associated with a particular representation into groups
based on their purpose. Each &lt;fileGrp&gt; must contain at least one
&lt;file&gt; belonging to that USE category. Values have not been
included in this vocabulary for services that deliver access to an
object because it is assumed that these would not be included in a
Submission Information Package or Dissemination Information Package.
Such a service may be archived as an object in its own right, in which
case the &lt;fileGrp&gt; would have a USE attribute of "original". This
decision may be reviewed if a use case emerges that requires a service
USE attribute.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="fileSec4"
<br>
				<p>There may be a variety of different types of derivative
(thumbnail, view, examination, streaming, special delivery). Within
each of these types there may be multiple sub-types, e.g., two
different-sized thumbnails, one for result sets, one for display with
the descriptive metadata. At this stage it is assumed that the
derivative &lt;fileGrp&gt; will contain all of the derivative files
needed for the successful delivery of an object. Within the derivative
&lt;fileGrp&gt;, the &lt;file&gt; USE attribute may be used in order to
distinguish between the different types of derivatives. Values for the
&lt;file&gt; USE attribute must be drawn from a controlled vocabulary,
which must be specified in a sub-profile. The Behaviors section of the
sub-profile must also detail the different types of derivatives
included in the &lt;fileGrp&gt;, and associate them with appropriate
behaviours for each &lt;structMap&gt; &lt;div&gt; TYPE.
</p>

constituting a digital object to a repository, &lt;fileSec&gt; must
contain one &lt;fileGrp&gt; with a USE attribute of "original" if the

USE attribute of "master" if the object is a digital surrogate.
&lt;fileSec&gt; may also contain &lt;fileGrp&gt; elements for files
with other USE attributes.

&lt;fileGrp&gt; with the same USE attribute, however, it must be
distinguished from other &lt;fileGrp&gt; instances with the same USE
attribute through the use of the &lt;fileGrp&gt; VERSDATE attribute.

@ID="fileSec8"
<br>
				<p>Use of &lt;fileGrp&gt; ID and ADMID attributes is not supported by this profile.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="fileSec9"
<br>
				<p>&lt;file&gt; must contain ID, MIMETYPE, SIZE, CHECKSUM and
CHECKSUMTYPE attributes and either a single &lt;FLocat&gt; or
&lt;FContent&gt; element. The &lt;file&gt; ID attribute is for internal
use within the METS document (it pairs with IDREF in the &lt;fptr&gt;
FILEID attribute) but may be based on a repository identifier. The
&lt;file&gt; CHECKSUMTYPE attributes must contain values from the
controlled vocabulary specified in METS. This profile requires
MIMETYPE, SIZE, CHECKSUM and CHECKSUMTYPE to be recorded as METS
&lt;file&gt; attributes rather than as equivalent elements in a
&lt;techMD&gt; &lt;premisObject&gt; instance because providing this
information at the packaging level enables processing decisions to be
made without having to parse the package payload, and potentially
allows for a more modular design of processors. </p>
			</div>

@ID="fileSec10"
<br>
				<p>The &lt;file&gt; ADMID attribute must be used to provide a
link to the file's administrative metadata. ID values must be inserted
in the ADMID attribute for all &lt;techMD&gt;, &lt;sourceMD&gt;,
&lt;digiprovMD&gt; and &lt;rightsMD&gt; elements related to this file
included in the &lt;amdSec&gt; element. Each ID value must be separated
by a space.
</p>

@ID="fileSec12"

&lt;stream&gt;, &lt;transformFile&gt; and any nested &lt;file&gt;
elements is not supported by this profile.
</p>

			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="fileSec13"
<br>
				<p>&lt;file&gt; may contain an OWNERID attribute to provide a
unique identifier (including a URI) assigned to the file which may
differ from the URI used to retrieve the file. This will often be the

recommended for files which may be used by a 'root' file to reconstruct
or render an object.
</p>

@ID="fileSec14"
<br>
				<p>&lt;FLocat&gt; is not repeatable in this profile. A
&lt;FLocat&gt; must be provided for each file if the content of the
file is not embedded in &lt;FContent&gt;. &lt;FLocat&gt; must only be
used if and only if the URL can be guaranteed under normal conditions
(i.e. excluding network /connectivity issues) to be accessible. Any URL
either needs to point directly to a file or to a service which exposes
the file content to the requesting party.
</p>

values from the controlled vocabulary specified in METS. However, the
LOCTYPE value "other" and the OTHERLOCTYPE attribute must not be used
in this profile. The location of the resource pointed to must be stored
in the XLINK:HREF attribute as specified in METS.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="fileSec16"
<br>
				<p>&lt;FContent&gt; must be present if there is no
&lt;FLocat&gt;. As specified in METS, the content file must be either
Base 64 encoded and contained within the subsidiary &lt;binData&gt;
wrapper element, or consist of XML information and be contained within
the subsidiary &lt;xmlData&gt; wrapper element. The XLINK attribute is
mandatory but this profile has no recommendations.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="fileSec17"
<br>
				<p>Use of the ID and USE attributes for the &lt;FLocat&gt; and &lt;FContent&gt; elements is not supported by this profile.
</p>
			</div>
		</div>
		<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#structMap">structMap:</a><br>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap1"
<br>
				<p>The structural map is the heart of a METS document,
defining the hierarchical arrangement of a primary source document
which has been digitized. This profile regards the purpose of this
element group as facilitating the deconstruction of a digital object
for ingest or the reconstitution of a digital object for delivery or
presentation. </p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap2"
<br>
				<p>There must be at least one &lt;structMap&gt; element that
describes the structure of the whole object represented by the METS
document. For an object consisting of a single file there will be a
single &lt;div&gt; element. For an object whose structure is
hierarchical the structure should be encoded as a tree of nested
&lt;div&gt; elements. The first level &lt;div&gt; element must
represent the whole object. Lower level &lt;div&gt; elements must
represent parts of the object. </p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap3"
<br>
				<p>If there is more than one &lt;structMap&gt;, the
&lt;structMap&gt; TYPE attribute is mandatory, and values must be taken
from the Australian METS Profile &lt;structMap&gt; TYPE controlled
vocabulary. Use of &lt;structMap&gt; ID attribute is mandatory if you
have more than one &lt;structMap&gt; of the same TYPE. </p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap4"
<br>
				<p>This profile supports the use of the optional &lt;structMap&gt; LABEL attributes but makes no recommendations.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap5"
<br>
				<p>The &lt;div&gt; TYPE attribute is mandatory and values
must be taken from the Australian METS Profile &lt;div&gt; TYPE
controlled vocabulary. </p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap6"
<br>
				<p>This profile supports the use of the optional &lt;div&gt;
LABEL and ORDERLABEL attributes, which may be used to represent the
relationship between components and the whole object.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap7"
<br>
				<p>The &lt;div&gt; DMDID attribute must be present for the
first level &lt;div&gt; representing the whole object and must be
present for the lower level &lt;div&gt; elements if there is a
corresponding &lt;dmdSec&gt; for that part of the object.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap8"
<br>
				<p>The &lt;div&gt; ADMID attribute must be present for the
first level &lt;div&gt; representing the whole object and must be
present for the lower level &lt;div&gt; elements if there is any
corresponding administrative metadata for the representation described.
ID values must be inserted separated by a space for all &lt;techMD&gt;,
&lt;sourceMD&gt;, &lt;digiprovMD&gt; and &lt;rightsMD&gt; elements
related to this representation included in the &lt;amdSec&gt; element.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap9"
<br>
				<p>Use of the &lt;div&gt; ID, ORDER and CONTENTIDS attributes is not supported by this profile.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap10"
<br>
				<p>There must be at least one &lt;fptr&gt; in each
&lt;div&gt; that references a &lt;premisObject&gt; of TYPE="file". Each
&lt;fptr&gt; must contain a FILEID to point to a file in the METS
document.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap11"
<br>
				<p>Use of the &lt;fptr&gt; ID and CONTENTIDS attributes and
the child elements &lt;par&gt;, &lt;seq&gt; and &lt;area&gt; is not
supported by this profile.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap12"
<br>
				<p>&lt;mptr&gt; may be used to reference any child object or component that has its own METS document.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap13"
<br>
				<p>Use of the &lt;mptr&gt; ID and CONTENTIDS attributes is not supported by this profile.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="structMap14"
<br>
				<p>Use of the &lt;structLink&gt; and &lt;behaviorSec&gt; element groups is not supported by this profile.
</p>
			</div>
		</div>
		<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#multiSection">multiSection:</a><br>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="multiSection1"
<br>
				<p>All date values must be xsd:dateTime compliant, as specified in XML Schema: Datatypes at http://www.w3.org/TR/xmlschema-2.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="multiSection2"
<br>
				<p>There must be only one &lt;mdWrap&gt; element in each
parent element. All &lt;mdWrap&gt; elements must contain an MDTYPE
attribute with a value from the controlled list prescribed by METS. If
the value of MDTYPE is "other", the &lt;mdWrap&gt; element must also
contain an OTHERMDTYPE attribute. Any schema identified in MDTYPE or
OTHERMDTYPE must be included as an extension schema in this profile or
a sub-profile conformant with this profile. The &lt;mdWrap&gt; element
must also include data encoded in the specified schema in the
&lt;xmlData&gt; element.
</p>
			</div>
			<div class="gx"><a href="http://www.loc.gov/standards/mets/profile_docs/mets.profile.v1-2.html#requirement">requirement:</a>
@ID="multiSection3"
<br>
				<p>Use of the &lt;mdRef&gt; element is not supported by this profile.</p>
			</div>
		</div>
	</div>
